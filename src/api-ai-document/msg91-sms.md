
# MSG91 SMS OTP — Porting Guide

Reference for reimplementing CollarCheck phone OTP delivery in another environment (Node, new PHP service, etc.). Values and request shapes are taken from this codebase as of the documented sources below.

**Related app flow:** `POST /wapi/login` → OTP stored locally → SMS via MSG91 → `POST /wapi/login/verify-otp` verifies against **local DB**, not MSG91.

---

## 1. Source map

| Piece | Location |
|-------|----------|
| Primary send helper | `app/Helpers/general_helper.php` → `otpSend($mobile, $otp)` |
| Thin wrapper used by login | `app/Controllers/MainController.php` → `send_otp_new($mobile, $otp)` |
| Alternate send (returns full JSON) | `MainController::send_otp_mobile` |
| MSG91 verify (legacy) | `MainController::verify_otp_mobile` |
| Template ID constant | `app/Config/Constants.php` → `TEMPLATE_ID` |
| Auth key | Env: `AUTH_KEY_MSG` via `getenv('AUTH_KEY_MSG')` |
| Login OTP orchestration | `GeneralApi::send_common_otp` |
| Login OTP verify (local) | `GeneralApi::verify_common_otp` |

### Call chain (unified login)

```
POST /wapi/login  { uniqueId: phone }
  → GeneralApi::login_common
  → send_common_otp($uniqueId)
       1. Generate 6-digit OTP
       2. INSERT/UPDATE table `otp` (expiry = now + 10 minutes)
       3. MainController::send_otp_new($phone, $otp)
            → otpSend($mobile, $otp)
                 → GET MSG91 Send OTP API
```

Other callers of `otpSend` (same MSG91 contract):

- `app/Controllers/restaurants/AuthController.php`

## 3. Send OTP API (primary — implement this)

This is what `otpSend()` does. **Use this for parity with login SMS.**

### Request

| Field | Value |
|-------|--------|
| Method | `GET` |
| URL | `https://control.msg91.com/api/v5/otp` |
| Query string | See table below |
| Timeout | 15s |
| Body | None |

#### Query parameters

| Parameter | Required | Example | Notes |
|-----------|----------|---------|--------|
| `template_id` | Yes | `64edcc1ad6fc051e7c079f72` | MSG91 OTP template ID (DLT-approved SMS template) |
| `mobile` | Yes | `919876543210` | Digits only; app strips `+` then passes as-is |
| `otp` | Yes | `482913` | **Generated by your app**, not by MSG91 |
| `otp_length` | Yes | `6` | Fixed to 6 in this codebase |

Full example URL:

```
https://control.msg91.com/api/v5/otp?template_id=64edcc1ad6fc051e7c079f72&mobile=919876543210&otp=482913&otp_length=6
```

#### Headers

| Header | Value | Notes |
|--------|--------|--------|
| `Content-Type` | `application/json` | Even for GET |
| `authkey` | `{AUTH_KEY_MSG}` | Primary helper sends **no space** after colon: `authkey:YOUR_KEY`. Alternate `send_otp_mobile` uses `authkey: YOUR_KEY` (with space). Both patterns appear in code; MSG91 accepts the authkey header either way. |

### Mobile number rules (this app)

1. `trim` the value.
2. Remove all `+` characters: `str_replace("+", "", $mobile)`.
3. Do **not** auto-prefix country code in `otpSend`.  
   Callers are expected to pass a number that already includes country code as stored (e.g. `91…` for India).
4. Empty mobile or empty OTP → fail without calling MSG91.

### Success / failure handling (parity with `otpSend`)

1. Parse response as JSON.
2. **Success:** `response.type === "success"` → treat as sent (`true`).
3. **Curl/network error:** log (this app: `OTP CURL Error: …`) → `false`.
4. Any other JSON or missing `type` → `false`.

Typical success shape (MSG91; fields may vary by account):

```json
{
  "type": "success",
  "message": "OTP sent successfully",
  "request_id": "..."
}
```

Failure shapes vary; do not rely only on HTTP status — this app keys off `type === "success"`.

### Pseudocode (language-agnostic)

```text
function sendOtpSms(mobile, otp):
  mobile = replace(mobile, "+", "")
  if mobile empty OR otp empty:
    return false

  url = "https://control.msg91.com/api/v5/otp"
       + "?template_id=" + TEMPLATE_ID
       + "&mobile=" + mobile
       + "&otp=" + otp
       + "&otp_length=6"

  response = HTTP GET url
    headers:
      Content-Type: application/json
      authkey: AUTH_KEY_MSG
    timeout: 15s

  if network error:
    log error
    return false

  body = parse JSON(response)
  return body.type == "success"
```

### curl (manual test)

```bash
export AUTH_KEY_MSG='your_msg91_auth_key_here'
export MOBILE='919876543210'
export OTP='123456'

curl -s -G 'https://control.msg91.com/api/v5/otp' \
  --data-urlencode "template_id=64edcc1ad6fc051e7c079f72" \
  --data-urlencode "mobile=${MOBILE}" \
  --data-urlencode "otp=${OTP}" \
  --data-urlencode "otp_length=6" \
  -H 'Content-Type: application/json' \
  -H "authkey: ${AUTH_KEY_MSG}"
```

Expect JSON with `"type":"success"` when credentials, template, and number are valid.

### Node.js example

```js
async function otpSend(mobile, otp) {
  const clean = String(mobile).replace(/\+/g, '').trim();
  if (!clean || !otp) return false;

  const url = new URL('https://control.msg91.com/api/v5/otp');
  url.searchParams.set('template_id', process.env.MSG91_TEMPLATE_ID || '64edcc1ad6fc051e7c079f72');
  url.searchParams.set('mobile', clean);
  url.searchParams.set('otp', String(otp));
  url.searchParams.set('otp_length', '6');

  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), 15_000);

  try {
    const res = await fetch(url, {
      method: 'GET',
      headers: {
        'Content-Type': 'application/json',
        authkey: process.env.AUTH_KEY_MSG,
      },
      signal: controller.signal,
    });
    const detail = await res.json();
    return detail?.type === 'success';
  } catch (err) {
    console.error('OTP CURL Error:', err.message);
    return false;
  } finally {
    clearTimeout(timer);
  }
}
```

## 4. Verify OTP API (legacy — optional)

Present in `MainController::verify_otp_mobile` but **not** used by:

- `POST /wapi/login/verify-otp` (`verify_common_otp`)
- Primary login flow

Unified login verifies OTP against the **local `otp` table** only.

### Request (if you need MSG91-side verify)

| Field | Value |
|-------|--------|
| Method | `GET` |
| URL | `https://api.msg91.com/api/v5/otp/verify` |

#### Query parameters

| Parameter | Value in this code |
|-----------|--------------------|
| `authkey` | `getenv('AUTH_KEY_MSG')` (query string, not header) |
| `mobile` | `'91' + $mobile` (hardcoded India `91` prefix) |
| `otp` | OTP string |

Example:

```
https://api.msg91.com/api/v5/otp/verify?authkey=YOUR_KEY&mobile=919876543210&otp=482913
```

Returns decoded JSON object on success, or `false` on curl error. No strict `type === success` check in this method.

> **Caution:** `verify_otp_mobile` always prefixes `91`. If `$mobile` already includes `91`, you can double-prefix. Prefer the **local DB verify** model used by login for parity.

## 5. App-side OTP lifecycle (needed for full login parity)

MSG91 only **delivers** the SMS. The app owns generation, storage, and verification for common login.

### Generate (before send)

```php
$otp = substr(rand(1000000, 9999999), 0, 6); // 6 digits
```

### Store table: `otp`

| Column | Example / rule |
|--------|----------------|
| `phone` | Phone uniqueId when phone path |
| `email` | Email uniqueId when email path (email uses SMTP, not MSG91) |
| `otp` | 6-digit string |
| `status` | `1` |
| `expiry` | `strtotime(now + 10 minutes)` (unix) |
| `create_date` | `Y-m-d H:i:s` |

Upsert: if a row for that phone/email exists → **update**; else **insert**.

### Verify (login — local only)

1. Load `otp` by phone (or email).
2. Reject if missing → invalid phone/email.
3. Reject if `expiry < now` → `"Otp Expired !"`.
4. Reject if `otp` mismatch → `"Invalid OTP!"`.
5. On success: issue session token, delete OTP rows.

### Bypass / QA (no MSG91 call)

| Mechanism | Behavior |
|-----------|----------|
| `bypass($uniqueId)` (`uri_helper.php`) | Listed test phones/emails → `send_common_otp` returns true without SMS |
| `user.register_type == 'qa'` | `login_common` returns success without calling send |
| Verify path QA/bypass | Skips OTP match and issues token |

Do not depend on MSG91 for these accounts.

## 6. Alternate PHP sender (same API, different return)

`MainController::send_otp_mobile`:

- Same MSG91 send URL and params as `otpSend`.
- Returns **full decoded JSON** (or `false` on curl error), not a boolean.
- Optionally updates some table OTP column when `$id` is passed (legacy).

For new environments, implement the **boolean `otpSend` contract** unless you need the raw payload.

## 7. Checklist for a new environment

- [ ] Create/reuse MSG91 account with OTP API access  
- [ ] Copy **Auth Key** into `AUTH_KEY_MSG` (or equivalent secret store)  
- [ ] Use or recreate SMS OTP **template** with ID `64edcc1ad6fc051e7c079f72`, or create a new template and update `TEMPLATE_ID` (India DLT registration may be required)  
- [ ] Implement **Send OTP** GET as in §3  
- [ ] Generate 6-digit OTP in app and pass as `otp` query param  
- [ ] Normalize mobile: strip `+`, include country code (e.g. `91…`)  
- [ ] Store OTP + 10-minute expiry; verify locally for login parity  
- [ ] Timeout 15s; treat success only when `type === "success"`  
- [ ] Do **not** commit secrets  
- [ ] Smoke-test with curl before wiring production traffic

## 8. What this integration is / is not

| Is | Is not |
|----|--------|
| MSG91 **OTP v5** send API | WhatsApp |
| SMS via registered **template_id** | Push notification |
| App-generated OTP passed to MSG91 | MSG91-generated OTP only (MSG91 can generate; this app supplies OTP) |
| Auth via `authkey` header on send | OAuth / JWT to MSG91 |
| Local DB verification for `/wapi/login/verify-otp` | Mandatory MSG91 verify for login |
