CollarCheck database — simple relationship diagrams

Same style as a classic ER diagram: tables, columns, PK/FK, and crow’s-foot lines. Split by topic so each picture stays readable. Soft links today (IDs stored, no MySQL FK constraints yet).

Important: companies and employees share one table — cyb_user (user_type 1 = employee, 2 = company). Columns named company / company_id → cyb_user.id

Big picture — how the product hangs together

One hub table (cyb_user) owns almost everything. Read lines as: one parent → many children. Junction tables sit in the middle for many-to-many.

One → many One → one (or optional one) Gold border = hub table FK = foreign key column (logical)

How to read this

  • 1 user (employee) has many experiences, skills, educations, job applications.
  • 1 user (company) has many jobs, benefits, galleries; also appears as company on experiences and jobs.
  • Junction tables (application, user_skill, company_benefits, follow) connect two sides — that is many-to-many.

User & profile (1 : many children)

Profile side tables hang off cyb_user. Lookups (city, designation, …) are shared master lists.

V2 tip

  • Add real FKs: user_details.user_id → cyb_user.id, etc.
  • Consider splitting employee vs company into separate tables (or strict checks on user_type).

Employment & company reviews

Core product graph: employee works at company → experience → company writes a review (rating).

Cardinalities

  • Employee 1 → N experiences
  • Company 1 → N experiences (as employer)
  • Experience 1 → N ratings/reviews
  • Rating 1 → N history rows

Jobs & applications (many-to-many)

Company posts jobs. Employees apply via cyb_application (junction). Collaborators help manage a job.

Many-to-many

  • cyb_application = Job ↔ Employee (like cart_items between cart and product).
  • V2: UNIQUE(job, user) + FKs to both tables.

Skills, languages & education

Skills and languages are many-to-many (junction tables). Education is one-to-many from the user.

Many-to-many

  • cyb_user_skill links user ↔ skill (like cart_items).
  • cyb_user_language links user ↔ language.
  • V2: unique pair + FKs on each junction.

Company extras (benefits, wishlist, connection)

Features that hang off a company user: benefits (N:M), gallery, wishlist, connection to employees.

Social, messages & notifications

Users follow users, message each other, and receive notifications — all pointing back at cyb_user.
